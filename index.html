Mesh-MVP-Chat

Headless ESP32 Captive Portal Chat Server

Project Overview

The Mesh-MVP-Chat is a lightweight, standalone communication hub designed for the ESP32. It functions as a localized social mesh, allowing any device with a browser to join a private network and communicate in real-time without internet access.

Networking

Built-in Captive Portal redirects all Wi-Fi traffic to the chat UI automatically.

Persistence

Messages and user nicknames are stored in the client's LocalStorage, surviving refreshes and reboots.

Monitoring

Real-time system telemetry (RAM, Uptime, Users) is broadcasted via WebSocket every 15 seconds.

System Architecture

The Mesh-MVP-Chat leverages the ESP32's capabilities to create a fully self-sufficient local network. Here’s a breakdown of how the different components interact.

1. The Initial Connection: Captive Portal

When a user connects to the ESP32's Wi-Fi network, the device automatically redirects them to the chat application hosted on the ESP32 itself.

2. Real-time Communication: WebSockets

Once connected, each client (phone) establishes a persistent WebSocket connection to the ESP32. This allows for instant, two-way communication.

3. Message Flow & Broadcasting

When a user sends a message, it's packed into a JSON object and sent to the ESP32. The ESP32 then broadcasts this message to all connected clients, ensuring everyone sees the conversation instantly.

4. System Information & User Management

The ESP32 continuously monitors its own resources and active connections, broadcasting updates to all clients in the 'System Info' channel and managing the 'Connected Users' list.

Key Technical Iterations

  • Dynamic Viewport: Implemented 100dvh and viewport-fit=cover to solve the "Android Navigation Bar" clipping issue.
  • User Management: Integrated a dynamic std::vector in C++ to track active nicknames and broadcast them to the UI.
  • JSON Relay: Utilizes ArduinoJson to package messages with sender metadata, allowing the UI to distinguish between incoming and outgoing bubbles.

Current C++ Logic Header

The core of the message routing is handled via the WebSocket event listener:

void onEvent(AsyncWebSocket *server, ...) {
    if (type == WS_EVT_CONNECT) {
        // Assign Client ID and broadcast join
        activeUsers.push_back({client->id(), "Guest"});
        broadcastSystemStatus();
    } 
    else if (type == WS_EVT_DATA) {
        // Relay JSON packets between nodes, handle special commands
        String msg = String((char*)data).substring(0, len);
        if(msg.startsWith("NAME_INIT:")) { /* update nickname */ }
        else { ws.textAll(msg); }
    }
}

Future Roadmap: LoRa Bridge

The next iteration involves bridging this Web UI to the Meshtastic LoRa network. This will allow the local Wi-Fi mesh to communicate with remote nodes over several kilometers using the Serial2 UART port.
